# Руководство по базе данных `inventory_v2` и `topo`

## 1. Что это за база

В рабочем комплекте есть две главные схемы:

- `inventory_v2` — входной слой. Здесь лежат каталоги объектов, профили оборудования, правила назначения профилей, сохраненные наборы выбора и снимки запуска.
- `topo` — выходной слой. Здесь лежат результаты конкретных расчетов: состав узлов, шкала времени, координаты, технические связи, лучи и покадровое состояние активных лучей.

Схема `inventory` нужна только для старой загрузки и миграции. Схема `topo_v2` служит запасным направлением выгрузки и в этом комплекте не является основной.

Главная цепочка данных выглядит так:

`inventory_v2 -> расчет сети -> topo`

## 2. Что читать для разных задач

| Задача | Что читать | Что это дает |
| --- | --- | --- |
| Понять, какие наборы объектов есть в базе | `inventory_v2.selection_catalog` и таблицы `*_catalog_member` | Готовые каталоги шлюзов, целей, спутников, HAPS, VSAT и теоретических абонентов |
| Получить паспорт конкретного объекта | Таблицы `inventory_v2.gateway`, `target`, `satellite`, `haps`, `vsat`, `theoretical_subscriber` | Исходные характеристики объектов до расчета |
| Понять, какие радиопрофили назначены объектам | Таблицы сущностей и таблицы `ground_terminal_profile`, `relay_payload_profile` | Какие параметры оборудования использует расчет |
| Восстановить пользовательский набор выбора | `inventory_v2.selection_profile` и `selection_profile_item` | Сохраненный вручную набор объектов |
| Повторить точный состав конкретного запуска | `inventory_v2.simulation_request_snapshot` и `simulation_request_snapshot_item` | Неизменяемый снимок набора объектов на момент запуска |
| Посмотреть список прогонов | `topo.simulation` | Перечень расчетов с конфигурацией и состоянием |
| Построить движение объектов по времени | `topo.node`, `topo.simulation_time`, `topo.node_position` | Координаты всех узлов по кадрам |
| Построить технический граф связей | `topo.edge` | Связи вида `Satellite -> Ray -> ...` |
| Построить картину активных лучей | `topo.ray` и `topo.connected_ray` | Паспорт луча и его покадровое состояние |
| Получить готовый снимок кадра целиком | `topo.topology_snapshot` | Готовый `JSON` со связями и активными лучами на конкретном шаге |

## 3. Как понимать происхождение данных

В этом руководстве происхождение данных делится на три типа:

- Исходные данные. Они приходят из старых каталогов, из заранее подготовленных наборов или из посевных таблиц до расчета сети.
- Производные входные данные. Они вычисляются до расчета сети: устранение дублей, назначение ключей, назначение профилей, построение каталогов выбора.
- Результаты расчета. Они появляются только после запуска модели и отражают уже рассчитанную топологию.

Если коротко:

- почти вся схема `inventory_v2` состоит из исходных и производных входных данных;
- почти вся схема `topo` состоит из результатов расчета.

## 4. Схема `inventory_v2`

### 4.1. Общие поля таблиц с сущностями

У таблиц `gateway`, `target`, `satellite`, `haps`, `vsat`, `theoretical_subscriber` есть повторяющиеся поля.

| Поле | Смысл | Когда появляется |
| --- | --- | --- |
| `*_id` | Внутренний числовой ключ строки | Создает база при записи |
| `*_key` | Устойчивый строковый ключ сущности | Вычисляется при перестроении `inventory_v2`; обычно совпадает с исходным `external_id`, а при конфликте получает суффикс из хеша |
| `fingerprint_sha1` | Служебный хеш нормализованного содержимого сущности | Вычисляется при перестроении `inventory_v2`; нужен для устранения дублей |
| `name` | имя | Приходит из исходного каталога |
| `metadata` | Дополнительные признаки и служебные сведения | Приходят из исходного каталога или добавляются при перестроении |

### 4.2. Каталоги выбора

#### `inventory_v2.selection_catalog`

Таблица описывает наборы выбора по типам сущностей.

| Поле | Смысл | Происхождение |
| --- | --- | --- |
| `catalog_id` | Внутренний ключ каталога | Создает база |
| `kind` | Тип сущностей: `gateways`, `targets`, `satellites`, `haps`, `vsats`, `theoretical_subscribers` | Заполняется при перестроении или создании каталога |
| `catalog_key` | Строковый ключ каталога | Приходит из старого `set_key` или задается при создании |
| `name` | Название каталога | Приходит из старых наборов или задается пользователем |
| `description` | Описание каталога | Приходит из старых наборов или задается пользователем |
| `source_type` | Источник каталога | Производное поле: например, ручной каталог или посевной каталог |
| `metadata` | Дополнительные сведения о каталоге | Производное поле |

#### Таблицы `*_catalog_member`

Это таблицы связей между каталогами и сущностями:

- `gateway_catalog_member`
- `target_catalog_member`
- `satellite_catalog_member`
- `haps_catalog_member`
- `vsat_catalog_member`
- `theoretical_subscriber_catalog_member`

Общий смысл полей:

| Поле | Смысл | Происхождение |
| --- | --- | --- |
| `catalog_id` | Каталог, в который входит объект | Берется из `selection_catalog` |
| идентификатор сущности | Конкретная строка сущности | Берется из таблицы сущностей |
| `ordinal` | Порядок внутри каталога | Приходит из исходного набора или задается пользователем |
| `metadata` | Служебные сведения о включении объекта в каталог | При перестроении или ручном сохранении |

Эти таблицы нужны, когда надо получить готовый набор объектов по одному ключу каталога.

## 5. Схема `topo`

### 5.1. Общий принцип

Почти каждая строка в `topo` относится к одному конкретному запуску, который определяется полем `simulation_uuid`.

Если задача связана с временем, почти всегда надо соединять таблицу с `topo.simulation_time` по паре:

- `simulation_uuid`
- `time_index`

### 5.2. `topo.simulation`

Запуск расчета.

| Поле | Смысл | Происхождение |
| --- | --- | --- |
| `simulation_uuid` | Уникальный ключ запуска | Создается при выгрузке результатов |
| `created_at` | Время записи запуска в базу | Ставит база |
| `name` | Название запуска | Приходит из сценария |
| `config` | Конфигурация запуска в `jsonb` | Формируется до записи результатов; это входные данные и производные входные данные, а не результаты |
| `status` | Состояние запуска | При записи результатов; обычно `finished` |

Поле `config` — это лучший источник, если надо понять, с какими настройками вообще выполнялся расчет.

### 5.3. `topo.simulation_time`

Шкала времени запуска.

| Поле | Смысл | Происхождение |
| --- | --- | --- |
| `simulation_uuid` | Запуск | Из `topo.simulation` |
| `time_index` | Номер шага | Вычисляется во время подготовки выгрузки |
| `time_msec` | Временная отметка шага в миллисекундах | Вычисляется из начала расчета и шага времени |

`time_msec` — это именно временная отметка шага, а не порядковый номер.

### 5.4. `topo.node`

Список узлов технического графа.

| Поле | Смысл | Происхождение |
| --- | --- | --- |
| `simulation_uuid` | Запуск | Из `topo.simulation` |
| `node_id` | Внутренний числовой ключ узла в рамках запуска | Формируется в расчетном слое |
| `node_uuid` | Уникальный строковый идентификатор узла в рамках запуска | Формируется при выгрузке; для обычных объектов берется из их идентичности, для синтетических узлов может вычисляться по `uuid5` |
| `node_type` | Тип узла: например, `CGS`, `Target`, `Satellite`, `HAPS`, `Ray`, `Region`, `VSAT`, `ImportantPlace` | Формируется при выгрузке |
| `name` | Имя узла | Приходит из исходных данных или строится при выгрузке |
| `net_id` | Внутренний номер подсети или группы | Формируется в расчетном слое |
| `config` | Паспорт узла в `jsonb` | Собирается из настроек исходного объекта и служебной идентичности |

`config` полезен, когда надо получить дополнительные признаки без отдельного соединения с входными таблицами. Если внутри `config` есть блок `identity`, его удобно использовать для связи с исходными ключами.

### 5.5. `topo.node_position`

Координаты узлов по времени.

| Поле | Смысл | Происхождение |
| --- | --- | --- |
| `simulation_uuid` | Запуск | Из `topo.simulation` |
| `node_id` | Узел | Из `topo.node` |
| `time_index` | Шаг времени | Из `topo.simulation_time` |
| `x`, `y`, `z` | Координаты в системе `ECEF`, километры | После расчета геометрии |

Это основная таблица для построения движения.

Практический смысл:

- статические объекты повторяются на каждом шаге времени;
- подвижные объекты меняют координаты по кадрам;
- лучи как отдельные точки здесь не хранятся.

### 5.6. `topo.edge`

Технические связи между узлами.

| Поле | Смысл | Происхождение |
| --- | --- | --- |
| `simulation_uuid` | Запуск | Из `topo.simulation` |
| `edge_id` | Внутренний ключ связи | Создает база |
| `time_index` | Шаг времени | После расчета |
| `begin_node_id`, `end_node_id` | Начальный и конечный узлы технической связи | После расчета |
| `edge_type` | Тип технической связи, например `Satellite-Ray`, `Ray-CGS` | После расчета |
| `is_bidirectional` | Признак двусторонней связи | После расчета |
| `config` | Дополнительные сведения о связи | Сейчас в основном служебное поле; часто пустое |

Важно: `topo.edge` хранит именно технический граф. Если нужен путь вида `Satellite -> Ray -> VSAT`, его надо брать отсюда. Если нужен уже готовый снимок активного кадра без разбора всех шагов, удобнее смотреть `topo.topology_snapshot`.

### 5.7. `topo.region`

Служебные корзины пространственного разбиения.

| Поле | Смысл | Происхождение |
| --- | --- | --- |
| `simulation_uuid` | Запуск | Из `topo.simulation` |
| `node_id` | Узел типа `Region` | После подготовки расчетной сцены |
| `region_code` | Числовой код корзины | Производное поле до расчета связи |

Эта таблица нужна главным образом для внутренней логики расчета. Для прикладного анализа обычно важнее конечные точки и лучи, а не сама корзина.

### 5.8. `topo.vsat`

Специализированные строки для расчетных конечных точек `VSAT`, которые были развернуты из корзин.

| Поле | Смысл | Происхождение |
| --- | --- | --- |
| `simulation_uuid` | Запуск | Из `topo.simulation` |
| `node_id` | Узел конечной точки | После подготовки расчетной сцены |
| `region_code` | Код корзины или региона | Из входных данных и пространственного разбиения |
| `la`, `lo` | Географическая широта и долгота | Из входных данных после нормализации |
| `alt` | Высотный параметр конечной точки в расчетной геометрии | Подготавливается до расчета связи |

Если для анализа нужна именно исходная предметная карточка терминала, надежнее брать ее из `inventory_v2.vsat`. Таблица `topo.vsat` нужна прежде всего для связи с расчетным графом.

### 5.9. `topo.ray`

Паспорт лучей.

| Поле | Смысл | Происхождение |
| --- | --- | --- |
| `simulation_uuid` | Запуск | Из `topo.simulation` |
| `node_id` | Идентификатор луча как отдельного узла графа | Формируется при построении сети |
| `origin_node_id` | Узел-источник луча | После построения сети |
| `ray_kind` | Вид луча | После построения сети |
| `cluster_id` | Внутренняя группа лучей у источника | После построения сети |
| `frequency_id` | Частотный канал или диапазон | После построения сети |
| `aperture` | Угловая ширина луча, радианы | После построения сети |
| `connect_limit` | Максимум одновременных соединений на луч | После построения сети |

Это справочная таблица. Она описывает луч как ресурс, но не говорит, что происходит с ним по времени.

### 5.10. `topo.connected_ray`

Покадровое состояние активных лучей.

| Поле | Смысл | Происхождение |
| --- | --- | --- |
| `simulation_uuid` | Запуск | Из `topo.simulation` |
| `ray_node_id` | Луч | Из `topo.ray` |
| `time_index` | Шаг времени | После расчета |
| `origin_node_id` | Узел-источник луча | После расчета |
| `origin_node_type` | Тип узла-источника | После расчета |
| `ray_kind` | Вид луча | После расчета |
| `cluster_id` | Внутренняя группа лучей | После расчета |
| `frequency_id` | Частотный канал | После расчета |
| `aperture` | Угловая ширина луча, радианы | После расчета |
| `connect_count` | Сколько соединений реально обслуживает луч на этом шаге | После расчета |
| `target_node_id` | Явная цель луча, если она определена | После расчета |
| `elevation` | Угол места, радианы | После расчета |
| `length` | Длина трассы, километры | После расчета |
| `alt` | Служебная величина из текущей реализации | После расчета |
| `beam_target_offset_rad` | Отклонение цели от центра диаграммы, радианы | После расчета |
| `pointing_x`, `pointing_y`, `pointing_z` | Вектор направления луча | После расчета |
| `k` | Старое служебное поле; сейчас почти всегда не используется | После расчета |

Очень важное замечание по `alt`:

поле `alt` в текущей реализации не стоит считать «высотой цели». По фактическому смыслу оно ближе к длине радиус-вектора до целевой точки. Для прикладного использования безопаснее опираться на:

- `length`,
- `beam_target_offset_rad`,
- `pointing_x`, `pointing_y`, `pointing_z`,
- координаты самого целевого узла из `topo.node_position`.

Если нужна картина активных лучей для визуализации и анализа нагрузки, это главная таблица.

### 5.11. `topo.topology_snapshot`

Готовый снимок кадра в `jsonb`.

| Поле | Смысл | Происхождение |
| --- | --- | --- |
| `simulation_uuid` | Запуск | Из `topo.simulation` |
| `time_index` | Шаг времени | После расчета |
| `snapshot` | Готовый снимок активной топологии на шаге | После расчета |

Это удобный слой, если не хочется заново собирать кадр из множества таблиц. Обычно в `snapshot` лежат:

- активные связи;
- активные лучи;
- сведения о связности;
- служебные признаки кадра.

## 6. Что лучше использовать под типовые задачи

### 6.1. Нужно выбрать исходные объекты для нового расчета

Читайте:

- `inventory_v2.selection_catalog`
- таблицы `*_catalog_member`
- таблицы сущностей

### 6.2. Нужно понять, какие профили реально применяются к объектам

Читайте:

- поля `ground_terminal_profile_key`, `user_beam_profile_key`, `feeder_beam_profile_key` в таблицах сущностей;
- затем соединяйте их с `ground_terminal_profile` и `relay_payload_profile`.

### 6.3. Нужно повторить точный состав старого запуска

Читайте:

- `inventory_v2.simulation_request_snapshot`
- `inventory_v2.simulation_request_snapshot_item`

Это надежнее, чем брать текущие каталоги, потому что каталоги со временем могут измениться.

### 6.4. Нужно построить движение всех объектов

Читайте:

- `topo.node`
- `topo.simulation_time`
- `topo.node_position`

### 6.5. Нужно понять, какие лучи реально работают

Читайте:

- `topo.ray`
- `topo.connected_ray`

### 6.6. Нужно восстановить технический граф расчета

Читайте:

- `topo.node`
- `topo.edge`

Здесь будут видны и промежуточные узлы типа `Ray`.

### 6.7. Нужно быстро отдать готовый кадр наружу

Читайте:

- `topo.topology_snapshot`

## 7. Примеры запросов

### 7.1. Показать каталоги спутников и их состав

```sql
SELECT
    c.catalog_key,
    s.satellite_key,
    s.name,
    m.ordinal
FROM inventory_v2.selection_catalog AS c
JOIN inventory_v2.satellite_catalog_member AS m
    ON m.catalog_id = c.catalog_id
JOIN inventory_v2.satellite AS s
    ON s.satellite_id = m.satellite_id
WHERE c.kind = 'satellites'
ORDER BY c.catalog_key, m.ordinal;
```

### 7.2. Показать траекторию одного узла по времени

```sql
SELECT
    t.time_index,
    t.time_msec,
    p.x,
    p.y,
    p.z
FROM topo.node_position AS p
JOIN topo.simulation_time AS t
    ON t.simulation_uuid = p.simulation_uuid
   AND t.time_index = p.time_index
WHERE p.simulation_uuid = :simulation_uuid
  AND p.node_id = :node_id
ORDER BY t.time_index;
```

### 7.3. Показать активные лучи с именами источника и цели

```sql
SELECT
    cr.time_index,
    src.name AS origin_name,
    dst.name AS target_name,
    cr.ray_kind,
    cr.connect_count,
    cr.length,
    cr.elevation
FROM topo.connected_ray AS cr
JOIN topo.node AS src
    ON src.simulation_uuid = cr.simulation_uuid
   AND src.node_id = cr.origin_node_id
LEFT JOIN topo.node AS dst
    ON dst.simulation_uuid = cr.simulation_uuid
   AND dst.node_id = cr.target_node_id
WHERE cr.simulation_uuid = :simulation_uuid
ORDER BY cr.time_index, cr.ray_node_id;
```
